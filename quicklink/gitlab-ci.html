    # GitLab CI/CD 备忘清单

    本备忘单总结了 GitLab CI/CD (opens new window) 常用的配置说明，以供快速参考

    # 入门

    # 介绍

    创建并运行您的第一个 GitLab CI/CD 管道，在开始之前，请确保您拥有：

    • GitLab 中您想要使用 CI/CD 的项目
    • 项目的维护者或所有者角色

    如果您没有项目，可以在 https://gitlab.com (opens new window) 上免费创建一个公共项目

    • 在存储库的根目录下创建一个 .gitlab-ci.yml 文件。该文件是您定义 CI/CD 作业的地方
    • 转到Settings > CI/CD 并展开运行程序，只要您至少有一个运行器处于活动状态，旁边有一个绿色圆圈，您就有一个运行器可以处理您的工作

    # 示例

  image: node:16

windows_job:
  only:
    - master
  tags:
    - windows
  script:
    - echo Hello, %USERNAME%!

linux_job: tags:
    - linux script:
    - echo "Hello, $USER!"

    # 关键字

    # 关键字

    关键字 描述
    default 工作关键字的自定义默认值 # (opens new window)
    include 从其他 YAML 文件导入配置 # (opens new window)
    stages 管道阶段的名称和顺序 # (opens new window)
    variables 为管道中的所有作业定义 CI/CD 变量 # (opens new window)
    workflow 控制运行什么类型的管道 # (opens new window)

    # 关键字

    关键字 描述
    after_script 覆盖一组在作业之后执行的命令 # (opens new window)
    allow_failure 允许作业失败。失败的作业不会导致管道失败 # (opens new window)
    artifacts 成功时附加到作业的文件和目录列表 # (opens new window)
    before_script 覆盖在作业之前执行的一组命令 # (opens new window)
    cache 应在后续运行之间缓存的文件列表 # (opens new window)
    coverage 给定作业的代码覆盖率设置 # (opens new window)
    dast_configuration 在作业级别使用 DAST 配置文件中的配置 # (opens new window)
    dependencies 通过提供要从中获取工件的作业列表来限制将哪些工件传递给特定作业 # (opens new window)
    environment 作业部署到的环境的名称 # (opens new window)
    except 控制何时不创建作业 # (opens new window)
    extends 此作业继承自的配置条目 # (opens new window)
    image 使用 Docker 镜像 # (opens new window)
    inherit 选择所有作业继承的全局默认值 # (opens new window)
    interruptible 定义作业是否可以在被较新的运行冗余时取消 # (opens new window)
    needs 在阶段排序之前执行作业 # (opens new window)
    only 控制何时创建作业 # (opens new window)
    pages 上传作业的结果以与 GitLab Pages 一起使用 # (opens new window)
    parallel 应并行运行多少个作业实例 # (opens new window)
    release 指示运行器生成释放对象 # (opens new window)
    resource_group 限制作业并发 # (opens new window)
    retry 发生故障时可以自动重试作业的时间和次数 # (opens new window)
    rules 用于评估和确定作业的选定属性以及是否创建的条件列表 # (opens new window)
    script 由运行器执行的 Shell 脚本 # (opens new window)
    secrets CI/CD 保密工作需要 # (opens new window)
    services 使用 Docker 服务映像 # (opens new window)
    stage 定义作业阶段 # (opens new window)
    tags 用于选择跑步者的标签列表 # (opens new window)
    timeout 定义优先于项目范围设置的自定义作业级超时 # (opens new window)
    trigger 定义下游管道触发器 # (opens new window)
    variables 在作业级别定义作业变量 # (opens new window)
    when 何时运行作业 # (opens new window)

    # workflow:rules:variables

    您可以在 workflow:rules 中使用变量来定义特定管道条件的变量

    variables:
  DEPLOY_VARIABLE: "default-deploy"

workflow:
  rules:
    - if: $CI_COMMIT_REF_NAME == $CI_DEFAULT_BRANCH
      variables:
        DEPLOY_VARIABLE: "deploy-production"  # 覆盖全局定义的 DEPLOY_VARIABLE
    - if: $CI_COMMIT_REF_NAME =~ /feature/
      variables:
        IS_A_FEATURE: "true"                  # 定义一个新变量
    - when: always                            # 在其他情况下运行管道

job1:
  variables:
    DEPLOY_VARIABLE: "job1-default-deploy"
  rules:
    - if: $CI_COMMIT_REF_NAME == $CI_DEFAULT_BRANCH
      variables:                                   # 覆盖定义的 DEPLOY_VARIABLE
        DEPLOY_VARIABLE: "job1-deploy-production"  # 在工作层面。
    - when: on_success                             # 在其他情况下运行作业
  script:
    - echo "以 $DEPLOY_VARIABLE 作为参数运行脚本"
    - echo "如果 $IS_A_FEATURE 存在则运行另一个脚本"

job2:
  script:
    - echo "以 $DEPLOY_VARIABLE 作为参数运行脚本"
    - echo "如果 $IS_A_FEATURE 存在则运行另一个脚本"

    # allow_failure:exit_codes

    控制何时允许作业失败。对于任何列出的退出代码，作业是 allow_failure: true，对于任何其他退出代码，allow_failure false

    - echo "退出代码 137 的脚本允许此作业失败"
